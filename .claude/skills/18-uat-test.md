# UAT 階段 - 客戶驗收執行

## 前置條件
- Alpha 修正完成，功能已部署至 PROD（或 Beta 環境）
- 已知本次驗收涵蓋的模組與對應報價單號

## 輸入資訊
- **本次驗收模組**（主平台 / 新增模組 / 追加功能）
- **對應報價單號**（用於驗收單 Items 欄位）
- **部署完成日期**（填入驗收單 Items 說明）

---

## 執行步驟

### Step 0｜SIT — SF 內部測試（在客戶 UAT 之前）

客戶 UAT 之前，SF 應在 Demo/Staging 環境先完成內部系統整合測試（SIT）：

1. **部署至 Demo 環境**，提供 Demo 網址
2. **逐功能執行 SIT 測試**，覆蓋本批次的所有頁面與流程
3. **產出 SIT 驗收文件**（命名：`{客戶}_{專案}_SIT驗收文件_v1_{日期}.docx`），按以下結構記錄：
   - **驗收說明**：Demo 網址 + 測試時間
   - **驗收功能範圍**：按批次（第一批/第二批...）列出所有功能名稱
   - **SIT 測試結果 — 電腦版**：逐批次，每個功能/狀態一張截圖，命名格式：`{功能} - {子功能/狀態}`（例：首頁 - Banner、職缺內頁 - 空值狀態）
   - **SIT 測試結果 — 手機版**：同上
   - 若測試環境有技術限制（如 LINE 分享被 IP 封鎖），需在對應截圖項下附文字說明原因
4. **若有缺失 → 修正 → 更新版本號**（v1 → v2... → 直到 clean）
5. SIT 通過後，才移交客戶執行正式 UAT

> 大型系統通常按批次（Phase/Batch）各自做 SIT，同一批次可能出 3-5 版 SIT 報告才達到乾淨標準。SIT 文件同時作為付款里程碑的交付佐證。

**Bug 追蹤（大型案適用）**：SIT 期間若發現缺失，建議用雙軌追蹤表管理：
- SF 內部 Bug ID（`#XXX`，循序編號）+ 客戶系統任務 ID（如 Azure DevOps TFS 編號）
- 嚴重度：1（重大/Critical）/ 2（中）/ 3（輕）
- 欄位：批次層（L1/L2/L3）/ 功能 / 區域 / 嚴重度 / 狀態 / SF 修復日 / 客戶確認日
- Bug 命名格式：`{角色}_{專案}_{功能}_{描述}`（例：`前端_iMatch_首頁_Banner自動播放問題`）

### Step 0.5｜弱點掃描（企業客戶適用）

部分企業客戶（金融/醫療/政府等）會在正式驗收前要求執行**弱點掃描（Vulnerability Scan）**：

1. **確認掃描工具**：客戶指定（如 SonarQube、Checkmarx）或使用 SF 標準工具
2. **執行掃描**，產出兩份獨立報告：
   - **Security Hotspots 調整報告**（資安敏感類：Hard-coded passwords / 弱加密 / Regex DoS 等）
   - **Issue List (Bug) 修復報告**（程式碼品質類：CSS 無效單位 / 重複屬性 / Promise 誤用等）
3. 每份報告格式：
   - **Executive Summary 表**：類別 / 規則名稱 / 數量 / 處理狀態 / 備註
   - **逐項說明**：掃描結果 + 數量 + 狀態 + 技術說明 + 處置
   - 處理狀態三種：`已調整 (Fixed)` / `誤判 (False Positive)`（需附技術說明與佐證）/ `未處置 (Won't Fix)`（需說明接受原因）
4. 若某項屬客戶 IT 既有程式碼，SF 說明風險後**由客戶 IT 決定是否調整**（不強制修改）
5. 客戶 IT/資安審查放行後，才進入最終驗收

> 弱點掃描通常在最後一次 SIT 完成後執行，是最終付款的前置條件。若客戶要求，需在 SOW 交付物中列入「弱掃調整報告」。

### Step 1｜準備測試清單（Google Sheets）
建立 Google Sheets，依功能流程逐項列出測試項目：

```
頁面/功能 | 測試步驟 | 預期結果 | 測試結果（Pass/Fail）| 備註
```

- 建立「前台」與「後台」分頁
- 每個主要流程一個分區（對應 UI 流程編號）
- 測試項目粒度：一個操作步驟一列
- 完成後將連結放入驗收單 No.附件項目

### Step 2｜產出驗收單（Customer Acceptance Form）

驗收單格式（依百靈佳案標準）：

```
Customer Acceptance Form

Project Name:  {專案名稱}
Customer:      {客戶名稱}
Version 1.0.0  Issued {YYYY-MM-DD}

No. | Items                        | 報價單 / 採購單號 | Check | Approval
----|------------------------------|-------------------|-------|----------
 1  | {模組名稱}                    | {報價單號}        |       |
    | {交付範疇條列說明}            |                   |       |
    | 本機制已於{日期}更新{環境}    |                   |       |
 2  | 附件：平台測試驗收             |                   |       |
    | <前台> {Google Sheets URL}    |                   |       |
    | <後台> {Google Sheets URL}    |                   |       |

Customer Approvals:
SugarFun Project Manager: Signature___ Name___ Date___
{客戶代表}:               Signature___ Name___ Date___
```

> 驗收單為英文格式（Customer Acceptance Form），Items 用中文說明
> 每個重大模組/里程碑發一份獨立驗收單（不合併在一份）
> 每份驗收單對應一張或多張報價單號

### Step 3｜客戶自行測試（約 2 週）
- 提供驗收單草稿與測試清單 Google Sheets 給客戶
- SF 支援期間回應問題，不主動測試代替客戶
- 區分反饋類型：
  - **驗收問題**：原範疇內的 bug / 功能不符 → 進入 beta-fix
  - **新需求（CR）**：超出原報價範疇 → 另開追加報價流程

### Step 4｜收集簽核
- 客戶確認測試通過後，簽署驗收單（書面或 email 確認+掃描回傳）
- SF PM 簽署，存檔為 PDF
- 命名規範：`{客戶}_{專案}驗收單_{模組}_{YYYYMMDD}.pdf`
