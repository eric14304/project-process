# 實作進行階段 - UX Mockup

## 工具與產出規格

- **工具**：Figma（主要）
- **產出層級**：Wireframe → Mockup（視專案需求）
- **User Flow 圖**：視情境決定
  - 流程複雜、狀況多、易混淆 → 先產出 User Flow 再進入 wireframe
  - 簡易流程 → 不一定需要

---

## 執行步驟

1. **（視需要）產出 User Flow 圖**
   - 複雜流程、多狀態、跨角色流程優先產出
2. **Wireframe 逐頁/逐流程製作**
   - 依 SOW 功能範圍與時程分批交付
   - Figma 線上版即為最新版，UX 自行管理（無正式版號）
3. **內部 review**
   - 參與者：PM、UI、技術單位
   - 目的：同步資訊、確認功能方向、技術可行性討論
4. **客戶過稿會議（逐流程確認）**
   - 格式：逐功能流程編號逐條討論（1. 加入會員、2. 點數兌換…）
   - 客戶提問 → SF 當場決議或標記「待確認」
   - 會後產出：會議紀錄 + Next Step（分甲乙方）
5. **修改與補稿**
   - 無固定修改輪數上限
   - 時程不允許時：拆階段交付，依優先級排序修改
6. **定稿移交**
   - 重大修改：直接通知 UI / F2E / B2E，視規模決定是否更新 Figma
   - 輕微調整：口頭交接
   - Interaction notes：通常標注於 Figma，部分以口頭補充

---

## B2E 協作方式

- B2E 自行根據 UX 設計 API / DB schema
- 有欄位或資料結構問題時，B2E 主動與 UX / F2E 討論
- API 規格設計以 B2E 主導，UX 不介入技術細節

---

## 使用者研究（群訪）

適用於面向真實使用者的產品（B2C / 求職平台 / 消費者 App）：

| 類型 | 對象 | 時機 | 目的 |
|------|------|------|------|
| 深度訪談（impl-prep-interview） | 客戶 PM/IT/業務 | 實作前 | 確認功能需求與 scope |
| 群訪（UX 使用者研究） | 真實使用者（如 HR、求職者） | UX 進行中 | 驗證設計假設、挖掘痛點 |

群訪規範：
- **訪綱不提前提供受訪者**：確保當天反應真實，可提前寄問卷收集背景（非訪綱）
- **受訪者分層取樣**：不同子公司/部門流程差異大時，橫跨不同組織取樣
- **1對1 訪談需測試環境**：受訪者用電腦 + 測試帳號 + Webex 側錄
- **後期群訪轉型**：初期聚焦需求，後期可在同一會議中同時 review wireframe

## Scope Escalation

- UX 階段超工時通常不發生（前期 SOW / Kickoff 已 cut scope）
- 若發生：由 **PM** 面對客戶處理
