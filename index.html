提案階段
3 個 · 全部完整
01
✓ 完整
/proposal-interview
初步訪談 + 內部評估。收集客戶需求、預算與時程，產出訪談紀錄，召集內部團隊執行技術/設計/商業評估，確認承接決策後進入報價。
訪談紀錄 md 內部評估文件 承接決策 GitLab Issue
承接 → /proposal-quote 不承接 → 流程結束
02
✓ 完整
/proposal-quote
人力工時報價。依功能複雜度估算各角色工時（PM/UX/UI/F2E/B2E/DevOps），套用報價策略（正常/保守/競標），產出客戶版與內部版報價文件。
工時估算表 報價文件 內部版含毛利 付款條件
→ 下一步：/proposal-pitch
03
✓ 完整
/proposal-pitch
向客戶提案/比稿。規劃競爭態勢與溝通策略，準備提案素材 Checklist，產出提案大綱與功能架構文件 v1，追蹤提案結果並產出覆盤文件。
提案策略規劃 功能架構文件 v1 Sitemap / PPT 提案覆盤
→ 得標後：/impl-prep-interview
實作準備階段
4 個 · 全部完整
04
✓ 完整
/impl-prep-interview
深度訪談。得標後多輪需求挖掘，產出角色矩陣、關係流程圖、會議紀錄、客戶交付物追蹤表、API 整合準備矩陣與需求確認文件。支援群訪規劃。
角色矩陣 需求確認文件 API 整合矩陣 Out of Scope
→ 下一步：/impl-prep-sow
05
✓ 完整
/impl-prep-sow
製作 SOW 工作說明書（13條結構）。含功能範圍、非功能需求、驗收條件、付款里程碑、Change Control 工時池、保固期、SLA。支援企業客戶多輪協商流程。
SOW 13 條 工時池 CR 機制 版本歷史 企業協商
→ 下一步：/impl-prep-gantt
06
✓ 完整
/impl-prep-gantt
產生人力甘特圖。使用 MS Project 或 Excel，排入各角色進場順序、里程碑節點，預留客戶審核等待時間。範疇變更時更新版本號。
MS Project / Excel 週為粒度 里程碑 版本管理
→ 下一步：/impl-prep-kickoff
07
✓ 完整
/impl-prep-kickoff
Kickoff 會議（主要為內部）。確認溝通機制、站會節奏、工具（LINE/Email/Azure DevOps），建立共用進度追蹤表，產出會議紀錄與 Action Items。
Kickoff 議程 共用追蹤表 溝通規範 資安問卷
→ 平行執行：/impl-ux · /impl-b2e
實作進行階段
6 個 · 1 完整 · 5 草稿
08
✓ 完整
/impl-ux
UX Mockup 設計。Figma 工具，User Flow → Wireframe → Mockup，含內部 review、客戶過稿（逐流程編號確認）、修改迭代與定稿移交。支援群訪（使用者研究）流程。
Figma User Flow 客戶過稿 群訪規劃
→ 下一步：/impl-ui（串行）
09
🚧 草稿
/impl-ui
UI Design 視覺設計。依 UX Mockup 產出設計稿，含 Design System/元件規範，交付 F2E 切版。多場過稿會議（逐功能流程討論），圖庫授權確認。
Design System 多場過稿 圖庫授權 PDF 交付
→ 下一步：/impl-f2e
10
🚧 草稿
/impl-b2e
後端實作。DB Schema 設計、API 規格定義（交 F2E 參考），實作與單元測試、Code review，部署至 SIT 前確認。與 UX 平行可進行。
DB Schema API 設計 Swagger Code Review
→ 下一步：/impl-integration
11
🚧 草稿
/impl-devops
DevOps Infra 工程。規劃雲端/地端架構，建置 SIT/UAT/PROD 環境、CI/CD pipeline、SSL/防火牆安全設定，交付運維文件 runbook。
Infra 架構 CI/CD 環境分離 資安設定
→ 下一步：/sit-test
12
🚧 草稿
/impl-f2e
前端實作。依 UI 設計稿（流程 PDF）切版，API 串接，分批打包交付（source code + 程式清單 PPT + 文件 docx），版控追蹤 git hash ↔ 客戶 build ID。
分批打包 版控雙軌 DemoSite 驗收 加密交付
→ 下一步：/impl-integration
13
🚧 草稿
/impl-integration
整合串接迭代。前後端串接、第三方系統整合，依 Sprint 節奏迭代，管理 Bug 優先級（P0/P1/P2），確認 SIT 進場條件後觸發部署。
第三方整合 Sprint 迭代 Bug 追蹤 SIT 進場條件
→ 下一步：/sit-test
SIT 內部測試階段
3 個 · 全部草稿
14
🚧 草稿
/sit-report
產生測試/驗收報告。從 SOW 功能清單自動提取測試項目，產出測試案例（Happy Path + Edge Case），建立驗收報告模板，分配測試人員。
AI 輔助產出 測試案例 SOW 可追溯 驗收模板
→ 下一步：/sit-test
15
🚧 草稿
/sit-test
部署 SIT + 整合測試執行。SIT 部署（手動或 CI/CD）、Smoke test 驗證後，PM/Sales/Account 搭配 AI 輔助按測試案例執行，記錄並分類問題（P0/P1/P2），更新測試報告，移交開發修正。
SIT 部署 PM/Sales 執行 問題分類 GitLab Issues
有問題 → /sit-fix 通過 → /uat-deploy
16
🚧 草稿
/sit-fix
修正功能與 Bug。依優先級排序修正任務，修正後重新驗證（Regression 範圍），更新測試報告。修正後 → /sit-test 重新執行驗證，迭代直到通過。
優先級排序 Regression Test UAT 進場條件 迭代回圈
修正後 → /sit-test（迭代）
UAT 客戶驗收階段
3 個 · 1 完整 · 2 草稿
17
🚧 草稿
/uat-deploy
部署至 UAT 環境。確認 UAT 環境規格（VM/第三方正式環境/真實資料），執行部署，Smoke test，提供客戶存取資訊，確認驗收時程。
UAT 環境 客戶存取 Config 差異 客戶 IT 協調
→ 下一步：/uat-test
18
✓ 完整
/uat-test
客戶驗收執行。先完成 SF 內部 SIT（含弱點掃描），再移交客戶。產出測試清單 Google Sheets 與 Customer Acceptance Form，區分驗收問題與 CR，取得雙方簽核。
SIT 驗收文件 弱點掃描 Acceptance Form Bug 雙軌追蹤
→ 修正：/uat-fix CR：/proposal-quote 通過：/prod-deploy
19
🚧 草稿
/uat-fix
修正客戶反饋。處理驗收問題，修正後回 SIT 驗證再更新至 UAT，迭代至驗收通過。視需要執行弱點掃描與壓力測試，記錄已知問題清單。
雙環境驗證 弱點掃描 壓力測試 已知問題清單
→ 下一步：/prod-deploy
正式上線 Production 階段

20
🚧 草稿
/prod-deploy
部署至 PROD 環境。非尖峰時段執行，確認 DNS/憑證/防火牆（協同客戶 IT），正式服務串接驗證，上線驗收簽核，維護移交文件，進入保固期。
